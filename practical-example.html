BRDD in Practice: Implementation Example

In this article, we apply BRDD concepts to a real-world inventory management scenario using the modular Use Case structure.

USE CASE: Product Consumption (UC001)

The orchestrator that coordinates the logic for a user consuming an item from stock.

🛡️ Business Rules (Validations)

  • R001: Validate if the product ID exists in the active catalog.
  • R002: Validate if the available stock is greater than or equal to the requested quantity.

💾 State Setters (Side Effects)

  • SET001: Update the product's last_consumption_date.
  • SET002: Decrease the total_stock by the consumed amount.

🌐 External Effects (External Clients)

  • EXT001: Notify the Logistics API about the stock movement.
  • EXT002: Trigger an email alert if stock falls below the minimum threshold.

📄 Documentation Standard (.md)

This is the "Master Prompt" that guides the AI in implementing the code below.

### `UC_STOCK_001` - Product Consumption
> Manages the reduction of physical stock after a purchase.

#### 🛡️ Business Rules
- **`R001`**: Product must exist. -> "Product not found."
- **`R002`**: Sufficient stock available. -> "Insufficient stock."

#### 🛠️ Setters
- **`SET001`**: Update last_consumption_date.
- **`SET002`**: Decrease total_stock.

#### ⚡ Effects
- **`EXT001`**: Notify Logistics API.
- **`EXT002`**: Alert if stock is low.

Technical Implementation Example

The Use Case simply orchestrates these sub-groups using the ExecutionContext:

// Use Case Logic
class ConsumeProductUseCase {
    execute(input) {
        const ctx = new DefaultExecutionContext();
        
        // 1. Enrichment
        const data = this.enricher.enrich(ctx, input);
        
        // 2. Validation (Business Rules)
        this.validator.validate(ctx, data);
        
        if (!ctx.isValid()) return ctx;
        
        // 3. Execution (Setters & External Effects)
        this.client.execute(ctx, data);
        
        return ctx;
    }
}

BRDD na Prática: Exemplo de Implementação

Neste artigo, aplicamos os conceitos do BRDD em um cenário real de gerenciamento de estoque usando a estrutura modular de Caso de Uso.

CASO DE USO: Consumo de Produto (UC001)

O orquestrador que coordena a lógica de um usuário consumindo um item do estoque.

🛡️ Regras de Negócio (Validações)

  • R001: Validar se o ID do produto existe no catálogo ativo.
  • R002: Validar se o estoque disponível é maior ou igual à quantidade solicitada.

💾 State Setters (Efeitos Internos)

  • SET001: Atualizar a last_consumption_date do produto.
  • SET002: Decrementar o total_stock pelo valor consumido.

🌐 Efeitos Externos (Clientes Externos)

  • EXT001: Notificar a API de Logística sobre a movimentação.
  • EXT002: Disparar alerta de e-mail se o estoque cair abaixo do limite mínimo.

📄 Padrão de Documentação (.md)

Este é o "Master Prompt" que guia a IA na implementação do código abaixo.

### `UC_ESTOQUE_001` - Consumo de Produto
> Gerencia a redução de estoque físico após uma compra.

#### 🛡️ Regras de Negócio
- **`R001`**: O produto deve existir. -> "Produto não encontrado."
- **`R002`**: Estoque suficiente disponível. -> "Estoque insuficiente."

#### 🛠️ Setters
- **`SET001`**: Atualizar last_consumption_date.
- **`SET002`**: Decrementar total_stock.

#### ⚡ Efeitos
- **`EXT001`**: Notificar API de Logística.
- **`EXT002`**: Alertar se o estoque estiver baixo.
