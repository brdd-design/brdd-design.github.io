BRDD in Practice: Implementation Example

In this article, we'll apply BRDD concepts to a real-world inventory management scenario. The objective is to explicitly separate rules, enrichments, and side effects.

1. Mapping Business Rules

Before coding, the business team defines the journey:

UseCase: Consume Product (US001)
Rules:
- R001: Validate if product exists
- R002: Validate if there is enough quantity

Enrichments:
- ENR001: Fetch current product details from database

Effects:
- CE001: Update partner systems (Stock API)
- SET001: Set consumption date

2. Implementing Services

Using the single-responsibility principle, each step becomes a dedicated service:

// EnrichService
class FetchProductEnricher implements EnrichService {
    enrich(context, input) {
        // ENR001: Fetch data
        context.addEffect("ENR001");
        return db.getProduct(input.id);
    }
}

// ValidationService
class CheckQuantityValidator implements ValidateService {
    validate(context, data) {
        if (data.stock < data.requested) {
            context.addError("R002", "Insufficient quantity");
        }
    }
}

3. The Use Case (Orchestrator)

The Use Case simply orchestrates the mapped rules and reports the Context:

class ConsumeProductUseCase {
    execute(input) {
        const ctx = new DefaultExecutionContext();
        
        const data = this.enricher.enrich(ctx, input);
        this.validator.validate(ctx, data);
        
        if (!ctx.isValid()) return ctx;
        
        // Execute main action
        this.client.execute(ctx, data); // CE001
        
        return ctx;
    }
}

BRDD na Prática: Exemplo de Implementação

Neste artigo, vamos aplicar os conceitos do BRDD em um cenário real de gerenciamento de estoque. O objetivo é separar explicitamente regras, enriquecimentos e efeitos colaterais.

1. Mapeamento das Regras

Antes de codar, o time de negócios define a jornada:

UseCase: Consumir Produto (US001)
Regras:
- R001: Validar se o produto existe
- R002: Validar se há quantidade suficiente

Enriquecimentos:
- ENR001: Buscar detalhes do produto no banco

Efeitos:
- CE001: Atualizar sistemas parceiros (API de Estoque)
- SET001: Setar data de consumo

2. Implementando os Serviços

Usando o princípio da responsabilidade única, cada passo vira um serviço dedicado:

// EnrichService
class FetchProductEnricher implements EnrichService {
    enrich(context, input) {
        // ENR001: Fetch data
        context.addEffect("ENR001");
        return db.getProduct(input.id);
    }
}

// ValidationService
class CheckQuantityValidator implements ValidateService {
    validate(context, data) {
        if (data.stock < data.requested) {
            context.addError("R002", "Quantidade insuficiente");
        }
    }
}

3. O Caso de Uso (Orquestrador)

O Use Case simplesmente orquestra as regras mapeadas e devolve o Contexto:

class ConsumeProductUseCase {
    execute(input) {
        const ctx = new DefaultExecutionContext();
        
        const data = this.enricher.enrich(ctx, input);
        this.validator.validate(ctx, data);
        
        if (!ctx.isValid()) return ctx;
        
        // Execute main action
        this.client.execute(ctx, data); // CE001
        
        return ctx;
    }
}
