🚀 BRDD in Practice: Implementation Guide

In this article, we apply BRDD concepts to a real-world scenario—Assignment Submission—using the version 0.2.0 metadata approach to demonstrate how code becomes self-documenting and decoupled.

The Scenario: Assignment Submission

Imagine an education platform. When a student submits an assignment, we need to orchestrate the following steps:

  • R001: Check if they are enrolled.
  • R002: Check if the course is active.
  • R003: Check if the deadline hasn't passed.
  • EFF_DEACTIVATE: Deactivate previous submissions if they exist.
  • EFF_NOTIFY: Notify instructors about the new submission.

Step 1: The EnrichService (Data Preparation)

The EnrichService is the role responsible for gathering all external data (Database, APIs, Cache) needed so that business rules can decide without performing I/O.

class SubmissionEnrichService:
    """ROLE: EnrichService - Fetches data for the decision process."""
    def enrich(self, input_data: SubmitInput) -> EnrichedData:
        # Mocking DB/API calls
        return EnrichedData(
            student_id=input_data.student_id,
            course_active=True,
            is_enrolled=True,
            deadline=datetime(2026, 12, 31),
            previous_submissions=["sub_001"]
        )

Step 2: The ValidateService (Declarative Mode)

Using the 0.2.0 library, the ValidateService uses metadata (@rule) to define what must be checked. The framework handles registering failures in the context.

class SubmissionValidateService(MetadataValidatorMixin):
    """ROLE: ValidateService - Checks rules and populates the context."""
    
    @rule("R001", "Student is not enrolled in this course.")
    def check_enrollment(self, data: EnrichedData) -> bool:
        return data.is_enrolled

    @rule("R003", "The submission deadline has passed.")
    def check_deadline(self, data: EnrichedData) -> bool:
        return datetime.now() <= data.deadline

Step 3: The BusinessService (Execution & Effects)

The BusinessService handles actual state changes (writes) and triggers events. It is purely executive.

class SubmissionBusinessService:
    """ROLE: BusinessService - Persistence and effect triggers."""
    def save_new_submission(self, data: EnrichedData, content: str) -> str:
        new_id = f"sub_{uuid.uuid4().hex[:6]}"
        return new_id

    def notify_instructors(self, submission_id: str):
        # Notification logic...
        print(f"Notifying about {submission_id}")

Step 4: The UseCase (The Flow Maestro)

The UseCase connects the roles. With the @use_case decorator, it gains a unique identity in the ecosystem and automates the creation of the final report.

@use_case("UC_EDU_001")
class SubmitAssignmentUseCase(MetadataUseCaseMixin):
    """ROLE: UseCase - Orchestrates the flow using specialized services."""
    def __init__(self, enrich, validate, business):
        self.enrich = enrich
        self.validate = validate
        self.business = business

    def execute(self, input_data: SubmitInput) -> ExecutionContext:
        # 1. Automatically create context via metadata
        context = self.create_context(input_data)
        
        # 2. Enrichment
        data = self.enrich.enrich(input_data)
        
        # 3. Declarative Validation
        self.validate.validate_rules(context, data)
        if not context.is_valid():
            return context
            
        # 4. Business Action
        new_id = self.business.save_new_submission(data, input_data.content)
        
        context.add_effect("EFF_NOTIFY")
        self.business.notify_instructors(new_id)
        
        return context

The Final Result: The "Audit Report"

At the end of execution, BRDD delivers a JSON object that serves as a complete audit of the transaction.

{
  "status": 201,
  "errors": [],
  "metadata": {
    "use_case": "UC_EDU_001",
    "rules_passed": ["R001", "R002", "R003"],
    "setters": ["SET_ID:sub_7a2b91"],
    "effects": ["EFF_NOTIFY"]
  }
}
