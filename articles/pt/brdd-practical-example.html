🚀 BRDD na Prática: Guia de Implementação

Neste artigo, aplicamos os conceitos do BRDD a um cenário do mundo real—Envio de Atividade—utilizando a abordagem de metadados da versão 0.2.0 para demonstrar como o código se torna autodocumentado e desacoplado.

O Cenário: Envio de Atividade (Assignment Submission)

Imagine uma plataforma de educação. Quando um aluno envia uma atividade, precisamos orquestrar os seguintes passos:

  • R001: Verificar se ele está matriculado.
  • R002: Verificar se o curso está ativo.
  • R003: Verificar se o prazo final não expirou.
  • EFF_DEACTIVATE: Desativar envios anteriores, se existirem.
  • EFF_NOTIFY: Notificar os instrutores sobre o novo envio.

Passo 1: O EnrichService (Preparação de Dados)

O EnrichService é o papel responsável por reunir todos os dados externos (Banco, APIs, Cache) necessários para que as regras de negócio possam decidir sem realizar I/O.

class SubmissionEnrichService:
    """ROLE: EnrichService - Busca dados para o processo de decisão."""
    def enrich(self, input_data: SubmitInput) -> EnrichedData:
        # Aqui você faria chamadas ao DB ou APIs
        return EnrichedData(
            student_id=input_data.student_id,
            course_active=True,
            is_enrolled=True,
            deadline=datetime(2026, 12, 31),
            previous_submissions=["sub_001"]
        )

Passo 2: O ValidateService (Modo Declarativo)

Utilizando a biblioteca 0.2.0, o ValidateService usa metadados (@rule) para definir o que deve ser verificado. O framework cuida de registrar as falhas no contexto.

class SubmissionValidateService(MetadataValidatorMixin):
    """ROLE: ValidateService - Verifica regras e popula o contexto."""
    
    @rule("R001", "Aluno não está matriculado neste curso.")
    def check_enrollment(self, data: EnrichedData) -> bool:
        return data.is_enrolled

    @rule("R003", "O prazo de envio já expirou.")
    def check_deadline(self, data: EnrichedData) -> bool:
        return datetime.now() <= data.deadline

Passo 3: O BusinessService (Execução e Efeitos)

O BusinessService lida com as mudanças de estado reais (escrita) e disparos de eventos. Ele é puramente executivo.

class SubmissionBusinessService:
    """ROLE: BusinessService - Persistência e gatilhos de efeitos."""
    def save_new_submission(self, data: EnrichedData, content: str) -> str:
        new_id = f"sub_{uuid.uuid4().hex[:6]}"
        return new_id

    def notify_instructors(self, submission_id: str):
        # Lógica de notificação...
        print(f"Notificando sobre {submission_id}")

Passo 4: O UseCase (O Maestro do Fluxo)

O UseCase conecta os papéis. Com o decorador @use_case, ele ganha uma identidade única no ecossistema e automatiza a criação do relatório final.

@use_case("UC_EDU_001")
class SubmitAssignmentUseCase(MetadataUseCaseMixin):
    """ROLE: UseCase - Orquestra o fluxo usando serviços especializados."""
    def __init__(self, enrich, validate, business):
        self.enrich = enrich
        self.validate = validate
        self.business = business

    def execute(self, input_data: SubmitInput) -> ExecutionContext:
        # 1. Cria o contexto automaticamente via metadados
        context = self.create_context(input_data)
        
        # 2. Enriquecimento
        data = self.enrich.enrich(input_data)
        
        # 3. Validação Declarativa
        self.validate.validate_rules(context, data)
        if not context.is_valid():
            return context
            
        # 4. Efeitos de Negócio
        new_id = self.business.save_new_submission(data, input_data.content)
        
        context.add_effect("EFF_NOTIFY")
        self.business.notify_instructors(new_id)
        
        return context

O Resultado Final: O "Relatório de Guerra"

Ao final da execução, o BRDD entrega um objeto JSON que serve como auditoria completa da transação.

{
  "status": 201,
  "errors": [],
  "metadata": {
    "use_case": "UC_EDU_001",
    "rules_passed": ["R001", "R002", "R003"],
    "setters": ["SET_ID:sub_7a2b91"],
    "effects": ["EFF_NOTIFY"]
  }
}
