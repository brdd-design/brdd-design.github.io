📜 Manifesto BRDD: As 4 Regras de Ouro

Business Rule Driven Design (BRDD) não é apenas uma estrutura de pastas; é um compromisso técnico fundamentado em quatro pilares inegociáveis. O objetivo é garantir que o Negócio e o Código operem sob a mesma gramática.

🏛 Os 4 Pilares

1. Codificação Única de Regras

Toda regra de negócio, validação ou efeito colateral deve ter um código identificador único (ex: R001). No BRDD, o código não é apenas lógica; é o elo direto com a documentação de negócio. Se uma regra muda no papel, o desenvolvedor sabe exatamente qual linha de código deve ser ajustada.

2. Narrativa de Contexto de Execução

Casos de Uso não devem retornar dados brutos; eles devem narrar o que aconteceu durante o processo através do ExecutionContext. A saída do sistema torna-se um relatório de execução: o que foi validado, o que foi alterado e quais efeitos externos foram disparados. Isso simplifica drasticamente o debugging e a auditoria de processos.

3. Especialização de Serviços

Para manter a manutenibilidade, a lógica deve ser dividida em papéis semânticos claros, evitando serviços com responsabilidades sobrecarregadas. No BRDD, cada serviço tem um propósito definido:

  • UseCase: Orquestrador do fluxo lógico.
  • ValidateService: Especialista em validar a integridade de uma regra.
  • EnrichService: Responsável por preparar o contexto de dados.
  • BusinessService: Responsável pela ação principal e efeitos colaterais.

4. Padrão de Resposta Unificado

Toda interação via API deve seguir um formato JSON estrito e previsível. Isso garante que qualquer consumidor—seja uma interface de usuário ou um agente de IA—saiba exatamente como interpretar o sucesso ou a falha de uma operação.

🤖 Arquitetura AI-First

O BRDD foi projetado especificamente para maximizar a eficiência de IAs e Copilots. Ao isolar regras em metadados e serviços especializados, eliminamos a ambiguidade técnica.

  • Master Prompts: A documentação em .md atua como uma especificação técnica "viva" e executável.
  • Segurança Arquitetural: Com fronteiras bem definidas, evitamos que a IA introduza efeitos colaterais indesejados durante a implementação de novas validações.
